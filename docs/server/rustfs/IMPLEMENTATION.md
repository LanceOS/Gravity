# RustFS Object Storage Architecture

RustFS is the internal object storage layer used in the Gravity backend to securely and efficiently manage document content and media attachments. 

Originally named for its initial local filesystem-based implementation, "RustFS" has been refactored to use an S3-compatible architecture backed by **MinIO**. This document outlines how RustFS works from infrastructure to the application service layer.

## 1. Infrastructure & Deployment

RustFS runs alongside the Gravity backend via Docker Compose.

- **Container Engine**: Uses the official `minio/minio` Docker image.
- **Service Name**: Defined as `rustfs` in the root `docker-compose.yml`.
- **Storage**: Data is persisted to a Docker volume (`gravity_rustfs_data`) which is mounted to `/data/rustfs` inside the container.
- **Networking**: The backend communicates with MinIO internally on port `9000` via the `gravity_internal` docker network.
- **Environment**: Configuration is injected via the root `.env` file (e.g., `RUSTFS_ENDPOINT`, `RUSTFS_ACCESS_KEY`, `RUSTFS_SECRET_KEY`, `RUSTFS_BUCKET`).

## 2. Core Abstraction (`server/src/lib/rustfs.ts`)

The raw interactions with MinIO are abstracted through the `RustFS` class, utilizing the standard `@aws-sdk/client-s3` library. 

### Bucket Prefix Structure
Because S3 utilizes flat buckets rather than actual folder hierarchies, `RustFS` simulates directories using "prefixes". 
Every note gets an isolated, deterministic prefix generated by `RustFS.getBucketPath()`:
`notes/{projectId}/{userId}/{noteUuid}/`

### Operations
The `RustFS` wrapper provides standard file-system-like methods:
- `saveFile()`: Wraps `PutObjectCommand`. If the root bucket does not exist, it automatically creates it.
- `readFile()` / `readFileUtf8()`: Wraps `GetObjectCommand`. Crucially, it catches `NoSuchKey` and `NotFound` errors and translates them to standard Node.js `ENOENT` error codes to simplify error handling downstream.
- `deleteFile()`: Wraps `DeleteObjectCommand`.
- `deleteBucket()`: Since S3 doesn't have real directories, this method uses `ListObjectsV2Command` to retrieve all keys under the prefix and deletes them sequentially.

## 3. Data Layer (`server/src/modules/notes/repositories.ts`)

The database interaction is explicitly split between relational metadata and object storage.

- **`MetadataRepository`**: Manages Postgres records via Drizzle ORM. It stores searchable metadata (titles, excerpts, timestamps, versions for optimistic locking) and the deterministic `bucketPath` linking to the RustFS object.
- **`NotesRepository`**: Serves as the domain-specific wrapper around `RustFS`. 
  - **`saveBody()` / `getBody()`**: Forces the document's main markdown content to always be saved as `body.md` within the prefix.
  - **`saveAttachment()` / `getAttachment()`**: Uploads user media alongside `body.md`. It explicitly guards against users overwriting `body.md`.

## 4. Service Layer (`server/src/modules/notes/services/notes.ts`)

The service layer orchestrates transactions between Postgres and RustFS to ensure consistency.

### Two-Phase Creation
When a note is created:
1. A new UUID is generated and the `bucketPath` is calculated.
2. The metadata is inserted into Postgres.
3. The markdown content is written to RustFS.
4. **Rollback**: If the RustFS write fails, the Postgres metadata record is deleted to prevent dangling empty notes.

### Optimistic Locking
During updates, `MetadataRepository.updateNoteMetadata` increments the note's `version` field. If two users edit the note concurrently, the second save fails on the Postgres constraint, preventing race conditions from blindly overwriting `body.md` in MinIO.

## 5. Maintenance & Cleanup (`server/src/jobs/cleanupOrphanedAssets.ts`)

Media attachments are heavily utilized, but users frequently delete images from their markdown text without deleting the actual file.

A background job handles cleanup:
1. It queries Postgres for all note `bucketPath`s.
2. It reads `body.md` from RustFS for each note.
3. It parses the markdown for `api/v1/notes/:noteId/media/:filename` regex patterns.
4. It compares the referenced files against a `ListObjectsV2Command` listing.
5. Any unreferenced files are systematically deleted from RustFS to reclaim storage space.
